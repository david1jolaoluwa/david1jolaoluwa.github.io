BUG 1 — NAVIGATION LINKS NOT WORKING
  Problem:
    Clicking navbar links (About Us, Tuition, Reviews, etc.) and the
    "Book a Free Consultation" button did not scroll to the correct
    section of the page.

  Root Cause:
    The site uses standard HTML anchor links (href="#section-id").
    When embedded inside Google Sites, the page runs inside an iframe.
    Browsers block anchor/hash navigation inside iframes for security
    reasons, so the links appeared to do nothing.

  Fix Applied:
    Replaced all anchor-based navigation with a JavaScript smooth scroll
    function. This intercepts every click on a "#" link and manually
    calculates the correct scroll position, accounting for the sticky
    navbar height. This method works correctly inside Google Sites iframes.

  Code added (near bottom of aiming_high_tutors.html):
    A <script> block with a DOMContentLoaded listener that attaches
    smooth scroll behaviour to all links with href starting with "#".

BUG 2 — "BOOK A FREE CONSULTATION" BUTTON NOT WORKING
  Problem:
    The hero button labelled "Book a Free Consultation" did not
    scroll down to the contact/booking form at the bottom of the page.

  Root Cause:
    Same iframe anchor-link issue as Bug 1 above.

  Fix Applied:
    Resolved automatically by the same JavaScript smooth scroll fix
    applied for Bug 1. No separate change was needed.

NEW FEATURE — FORM SUBMISSION DATABASE
  What was added:
    The contact/booking form now saves every submission to an online
    database so enquiries are never lost.

  How it works:
    - Uses JSONBin.io (free tier) as a simple cloud data store
    - When a visitor submits the form, JavaScript reads the existing
      submissions, appends the new one, and writes the updated list back
    - Each record stores: timestamp, name, email, phone, subject, message
    - On success, the form is replaced with a personalised thank-you message
    - On failure, an alert is shown and the form is restored so they can retry

  Setup required (one-time):
    See the SETUP INSTRUCTIONS section at the bottom of this file.

NEW FILE — ADMIN DASHBOARD (admin_dashboard.html)
  What it does:
    A private browser-based dashboard for viewing all form submissions.
    Not part of the public website — open it locally in any browser.

  Features:
    - Summary stats: total enquiries, today, this week, top subject
    - Full submissions table with date, name, email, phone, subject, message
    - Click any row to open a detail panel showing the full message
    - Search bar — filters rows live as you type
    - Sort by any column — click a column header to sort
    - Export to CSV — downloads all submissions as a spreadsheet
    - Delete individual entries
    - Credentials saved in the browser so you only enter them once
    - Setup instructions shown automatically if credentials are missing

VERSION 1.4 — BUG FIX: FORM SUBMISSION NOT SHOWING THANK-YOU MESSAGE
Date: Session 5
Files affected: aiming_high_tutors.html (edited)
---------------------------------------------------------------------

BUG DESCRIPTION:
  After filling in and submitting the contact form, the thank-you
  message did not appear and no confirmation was given to the user.

ROOT CAUSE:
  The original submission code used two fetch requests chained together:
    1. GET — read existing submissions from JSONBin
    2. PUT — write updated list back
  The first request was using the wrong response format. JSONBin returns
  data wrapped in a "record" key when fetched without the X-Bin-Meta:false
  header, but the code was not unwrapping it correctly, causing the
  chain to fail silently before reaching the thank-you message.

FIX APPLIED:
  - Added the X-Bin-Meta: false header to the GET request so JSONBin
    returns the raw data without the wrapper object
  - Updated the data parsing to correctly read the submissions array
  - Added HTTP status checks on both fetch calls so errors are caught
    immediately with a clear message rather than failing silently
  - Improved the error alert to show the specific error message so
    future issues are easier to diagnose

Date: Session 6
Files affected: aiming_high_tutors.html (edited), admin_dashboard.html (edited)
---------------------------------------------------------------------

BUG DESCRIPTION:
  Despite the v1.4 fix being applied, the form still did not show the
  thank-you message and the dashboard still failed to load submissions.
  No error alert appeared either, meaning it was failing silently.

ROOT CAUSE:
  Two separate but related issues were found:

  Issue 1 — aiming_high_tutors.html:
    The v1.4 fix had not been saved into the version of the file the
    client was using. The uploaded file still contained the unfixed code
    from v1.3. The fix was re-applied cleanly to the correct file.

  Issue 2 — admin_dashboard.html:
    The dashboard fetch request was missing the 'X-Bin-Meta: false'
    header, so JSONBin was returning data wrapped in a { record: { ... } }
    object. The code was trying to read data.record.submissions but the
    fixed website code writes data as { submissions: [...] } (no wrapper).
    This mismatch meant the dashboard could never read the submissions
    correctly even if they had been saved.

FIX APPLIED — aiming_high_tutors.html:
  - Entire Part 2 JS block rewritten cleanly from scratch
  - X-Bin-Meta: false header added to GET request
  - Specific HTTP status checks added (401 = bad key, 404 = bad bin ID)
  - Form now reads field values directly by ID rather than via form.querySelector
  - Basic validation added (name + email required before submit fires)
  - Thank-you message made slightly larger and more prominent
  - Error alert now shows the exact error code to help diagnose future issues

FIX APPLIED — admin_dashboard.html:
  - X-Bin-Meta: false header added to the loadData() fetch request
  - Data parsing updated from data.record.submissions to data.submissions
  - Specific 401 and 404 error messages added to the error handler

VERSION 1.6 — CRITICAL FIX: DATABASE METHOD REPLACED
Date: Session 7
Files affected: aiming_high_tutors.html (rewritten), admin_dashboard.html (rewritten),
                SETUP_GOOGLE_SHEETS.txt (created), CHANGELOG.txt (updated)
---------------------------------------------------------------------

BUG IDENTIFIED:
  A live server-side test was run against the JSONBin API using the
  client's actual Bin ID and API Key. The API returned:
    HTTP 403 — "Host not in allowlist"
  This means JSONBin was configured to only accept requests from
  specific approved domains. Since no domain was added to the allowlist,
  ALL requests were being rejected — including those from the website
  and the admin dashboard. This is why no data was ever saved and no
  thank-you message appeared.

  A second issue was also confirmed: JSONBin enforces CORS restrictions
  that block requests made from inside iframes (such as Google Sites
  embeds), meaning JSONBin could never work reliably for this use case
  even if the allowlist was cleared.

  MANUAL FIX FOR JSONBIN (if client wants to keep it):
    Log into jsonbin.io → Settings → CORS / Allowlist
    Either clear the allowlist entirely, or add:
      https://sites.google.com
    However this is NOT recommended — the iframe CORS issue remains.

DECISION:
  JSONBin replaced entirely with Google Sheets via Google Apps Script.

WHY GOOGLE SHEETS:
  - No allowlist or CORS restrictions — works from any domain
  - Works inside Google Sites iframes
  - Completely free with any Google account
  - Data stored in a familiar spreadsheet the client can open directly
  - Supports email notifications when new rows are added
  - No third-party account dependency beyond Google

CHANGES MADE — aiming_high_tutors.html:
  - Removed all JSONBin credentials (BIN_ID, API_KEY)
  - Replaced with a single SCRIPT_URL variable
  - Form submission now POSTs JSON to the Google Apps Script URL
  - Uses fetch() with mode: 'no-cors' — required for cross-origin
    requests from iframes. This means the response cannot be read,
    but the data IS received by the script. The thank-you message
    shows as soon as the request is sent.
  - Thank-you message is now larger and more prominent
  - Validation added: name and email required before submit fires

CHANGES MADE — admin_dashboard.html:
  - Removed all JSONBin logic (BIN_ID, API_KEY, jsonbin.io fetch calls)
  - Replaced credentials panel with single Apps Script URL input
  - Dashboard now reads data via GET request to the same script URL
    with ?action=read appended
  - Data returned as JSON array { submissions: [...] }

NEW FILE — SETUP_GOOGLE_SHEETS.txt:
  - Step-by-step plain English setup instructions
  - Full Apps Script code to copy and paste (both doPost and doGet)
  - doPost: receives form submissions and writes rows to the sheet
  - doGet:  returns all rows as JSON for the admin dashboard to read
  - Testing instructions
  - Notes on email notification setup

WHAT THE CLIENT NEEDS TO DO:
  Follow SETUP_GOOGLE_SHEETS.txt — approximately 5 minutes.
  Once done, paste the Apps Script Web App URL into:

VERSION 2.1 — DATABASE METHOD REPLACED: FORMSPREE
Date: Session 12
Files affected: index.html (rebuilt clean)
---------------------------------------------------------------------

ROOT CAUSE OF PERSISTENT 405/403 ERRORS:
  Google Apps Script deployments made through a Google Workspace
  account enforce domain-level restrictions automatically. The
  "Host not in allowlist" error was being triggered by Workspace
  Admin policy — not by the code itself. This affects all external
  requests regardless of the "Who has access" setting chosen at
  deployment time. The allowlist control in Google Workspace Admin
  is only available on higher tier plans than Business Starter.
  This could not be fixed without upgrading the Workspace plan.

SOLUTION — FORMSPREE:
  Replaced Google Apps Script entirely with Formspree.
  Formspree is a dedicated form backend service with no Google
  Workspace restrictions, no CORS issues, and no allowlist settings.

HOW FORMSPREE WORKS:
  - Form submits a standard POST request to formspree.io/f/xnjwonzd
  - Formspree receives the data and emails it to the account holder
  - All submissions are also visible at formspree.io/forms
  - No Google account required, no script deployment needed
  - Free plan: 50 submissions per month

FORMSPREE ACCOUNT DETAILS:
  Endpoint: https://formspree.io/f/xnjwonzd
  Dashboard: https://formspree.io/forms
  Plan: Free (50 submissions/month)

CHANGES MADE TO index.html:
  - Removed all Google Apps Script code and SCRIPT_URL variable
  - Removed all JSONBin code and credentials
  - Removed duplicate script block (file had two script sections)
  - Added single clean Formspree fetch with proper Accept header
  - Standard fetch (no no-cors needed — Formspree supports CORS)
  - Error handling restored with email fallback in alert message
  - All 11 verification checks passed before release
